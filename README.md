# Dice Card Game

Dice Card Game 是一個 2 到 4 人的線上房間制卡牌骰子遊戲。玩家先建立或加入房間，由房主開始遊戲；每位玩家從候選卡中選出 6 張、排列成 1 到 6 的位置，排列完成後卡牌會傳給右側玩家。回合中玩家擲骰，依骰子結果翻開自己收到的對應位置卡牌，處理卡牌效果並更新分數，直到遊戲進入結算。

## 遊戲概要

- 支援 2 / 3 / 4 人房間，房主可在至少 2 人時開始遊戲。
- 目前遊戲流程為 `waiting -> drafting -> arranging -> playing -> finished`。
- 每位玩家在選牌階段從完整 13 張牌中選出 6 張，剩下 7 張保留為補牌牌庫。
- 每位玩家在排牌階段調整 6 張卡的 1 到 6 位置。
- 所有玩家排牌完成後，卡牌會傳給右側玩家，並進入回合階段。
- 回合中由目前玩家擲骰，骰子結果對應自己收到的卡牌位置。
- 翻開並使用卡牌效果後，系統更新分數、卡牌狀態與下一位玩家。
- 玩家分數歸零或低於 0 時會被淘汰；遊戲依目前規則進入結算後顯示勝利者。

## 操作流程

1. 開始畫面
   進入網站後會先看到開始畫面，只顯示遊戲品牌與「開始」按鈕。按下「開始」後進入房間操作畫面。

2. 房間操作
   在 Lobby 輸入暱稱後可以建立房間，或輸入房間碼加入既有房間。暱稱會存在瀏覽器 `localStorage`，玩家識別存在目前分頁的 `sessionStorage`。成功建立或加入房間後，網址會帶上 `#room=房間碼`，重新整理時會嘗試回到該房間。

3. 準備畫面
   房間畫面會顯示房間碼、複製房間碼按鈕、玩家列表與玩家狀態。房主在房間仍是等待狀態時會看到「開始遊戲」按鈕；非房主或遊戲已開始時不會顯示可用的開始操作。房間至少需要 2 位玩家才能開始。準備畫面也提供離開房間按鈕與設定選單。

4. 選牌階段
   遊戲開始後，每位玩家會看到自己的完整 13 張牌。點選牌會立即選取，畫面會顯示已選數量，例如 2 / 6。選滿 6 張後，未選的 7 張會保留為補牌牌庫，玩家等待其他人完成選牌。所有玩家選滿後自動進入排牌階段。

5. 排牌階段
   玩家看到自己選出的 6 張卡，依序對應位置 1 到 6。這 6 張不是排給自己使用，而是排好後連同補牌牌庫傳給右側玩家。每張卡提供上移、下移控制，底部有位置摘要。未形成有效 6 張順序時送出按鈕停用；送出後該玩家的排序鎖定並等待其他玩家。所有玩家送出後，卡牌依規則傳給右側玩家並進入回合階段。

6. 回合階段
   桌面中央顯示目前回合、骰子與主要操作按鈕，玩家座位顯示頭像、名稱、分數與 1 到 6 的卡牌位置。輪到自己時可以擲骰；擲骰中按鈕顯示 loading 狀態，骰子播放動畫，結果落定後會指定要處理的位置。玩家需翻開該位置卡牌，接著在卡牌詳情中關閉或使用可用效果。

7. 卡牌詳情與效果
   已翻開的卡牌可以開啟詳情 modal。詳情顯示卡牌位置、來源、狀態、分數或效果描述。只有在自己的回合、處理自己剛擲出的目標卡牌、卡牌已翻開且尚未使用時，才會顯示「使用效果」操作。使用中會進入 loading 狀態，完成後卡牌標示為已使用。

8. 設定、回 Lobby 與離房
   房間右上角設定按鈕會開啟設定 modal，內容包含房間碼、玩家數與操作按鈕。「回到 Lobby」會離開目前房間並回到房間操作畫面；「離開房間」會離開目前房間並回到開始畫面；「取消」與「關閉設定」只關閉設定 modal。若最後一位玩家離房，房間會被清除。

## 多人桌面配置

- 2 人桌面：對手固定在上方，自己固定在下方，中間是骰子與回合操作區。
- 3 人桌面：兩位對手分別在左側與右側，自己在下方，中間保留桌面與骰子區。
- 4 人桌面：三位對手分別在左側、上方與右側，自己在下方。這是資訊密度最高的版面，卡牌列會壓縮以保留中央操作區。
- 所有桌面都會顯示分數摘要、目前回合提示、每位玩家的 1 到 6 位置卡牌，以及已翻開或已使用的卡牌狀態。

## 主要互動狀態

- Disabled：非目前玩家、房主以外的開始遊戲、未完成 6 張排序、已選滿 6 張候選卡、已送出排序、已淘汰、沒有可用卡牌或有同步操作進行中時，相關按鈕與卡牌操作會停用。
- Active：目前回合玩家、可點擊的目標卡牌、剛選取的候選卡、剛移動的排牌卡、開啟中的卡牌詳情或設定 modal 都會以 active 視覺狀態提示。
- Loading：建立房間、加入房間、開始遊戲、選牌、送出排牌、擲骰、翻牌與使用卡牌效果時，畫面會暫時停用互動並顯示進行中狀態。
- 擲骰：擲骰時中央骰子播放滾動動畫；結果確認後顯示點數，並提示要翻開的卡牌位置。
- 翻牌：只有目前玩家可翻開自己擲骰指定的位置。翻開後會顯示卡牌詳情，其他已翻開卡牌也可再次查看詳情。
- 使用卡牌效果：符合使用條件時，卡牌詳情會出現使用效果按鈕；使用後卡牌會標記為已用，該位置不能再次使用。
- 同步與還原：房間畫面會定期同步狀態；若網址包含有效 `#room=房間碼`，重新整理後會嘗試還原房間畫面。

## Local Development

需要 Node.js 20 或以上版本。

```bash
npm start
```

也可以使用：

```bash
npm run dev
```

本機服務預設網址：

```text
http://localhost:3000
```

前端靜態檔案位於 `Client/`，本機 HTTP 服務與靜態檔案由 `Server/server.js` 提供。

## Project Structure

- `Client/`: 瀏覽器端 UI、房間操作、遊戲流程渲染與 API client。
- `lib/rooms.js`: 房間建立、加入、離開、房主與開始遊戲規則。
- `lib/room-api.js`: 建立房間、加入、離開、開始、選牌、排牌、擲骰、翻牌與使用卡牌的 API handler。
- `lib/game-rules.js`: 遊戲 phase、選牌、排牌、傳牌、擲骰、翻牌、使用效果、淘汰與勝負判定。
- `lib/cards.js`: 卡牌資料與抽牌工具。
- `lib/public-view.js`: 依玩家產生可公開給前端的房間與遊戲視圖。
- `lib/stores.js`: Memory 與 Redis persistence adapters。
- `Server/server.js`: 本機 HTTP/static adapter。
- `api/rooms.js`: Vercel Function adapter。
- `scripts/`: smoke、API handler 與 game rules 檢查腳本。

## Vercel Deployment

專案可透過 `vercel.json` 與 `api/rooms.js` 部署到 Vercel。

Production Vercel deployments 需要 Redis-compatible persistent storage：

```text
UPSTASH_REDIS_REST_URL
UPSTASH_REDIS_REST_TOKEN
```

如果 production 環境沒有設定 Redis，`api/rooms.js` 會回傳明確的 `503`，避免多人遊戲狀態只存在單次 request 的記憶體中。

部署：

```bash
npx vercel
```

部署到 production：

```bash
npx vercel --prod
```
